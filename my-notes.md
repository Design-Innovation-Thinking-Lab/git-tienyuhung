# 我的 GitHub 學習筆記

## 💡 什麼是 Git 和 GitHub？

### Git（你的存檔神器）
Git 是一個「版本控制系統」。想像你在寫程式或文案，Git 可以幫你記錄每一次的修改（存檔點）。如果寫壞了，隨時可以「讀檔」回到過去。它也讓團隊合作變得超簡單，大家可以同時改同一個專案，最後再完美合併。

### GitHub（你的雲端工作站）
GitHub 就是把 Git 的功能搬到網路上，並配上一個好用的網頁介面。它是全球開發者的聚集地，無論你是要寫遊戲還是做數據分析，都能在這裡找到工具和志同道合的人。

---

## 🔄 搞懂 GitHub 的工作流程

GitHub flow 是一套輕量的工作邏輯，讓你可以在不破壞原本成果的前提下，大膽嘗試新想法。

### Repository（儲存庫/專案夾）
簡稱「Repo」。這就是你放專案的地方，像是一個雲端資料夾，記錄了專案裡所有的檔案和變更歷史。

### Cloning（複製到電腦）
GitHub 上的檔案是在雲端 ☁️。透過「Clone」，你可以把整個專案下載到自己電腦裡，用你喜歡的編輯器改完後，再同步回雲端。

### Committing & Pushing（存檔與上傳）

- **Commit（存檔點）：** 當你改好一段代碼，就建立一個 Commit，並寫下簡短說明（例如：「修正了登入按鈕的顏色」）。
- **Push（推上去）：** 把你在電腦上做好的這些「存檔點」推送到 GitHub 雲端，這樣夥伴才能看到你的最新進度。

## 💻 必須知道的 GitHub 術語

### README（說明書）
每個專案都該有一個 README 檔案，用來告訴別人這個專案是幹嘛的、該怎麼用。

### Branches（分支）
如果你想嘗試新功能又怕弄壞原本可以跑的程式，就開一個「分支」吧！這就像是在平行時空做事，等實驗成功了，再合併回主線（Main）。

### Forks（分叉/複製一份）
如果你看到別人的優秀專案想拿來修改，可以點「Fork」。這會在你的帳號下複製一份一模一樣的專案，你怎麼改都不會影響到原作者。

### Pull Requests（合併請求，簡稱 PR）
當你在分支改好了，想把成果合併回主線（或貢獻給別人的專案），就發一個 PR。大家可以在上面討論、檢查你的程式碼，沒問題後再點擊合併。

### Issues（問題追蹤/任務清單）
用來記錄專案中有什麼 Bug 要修、有什麼新功能要做。它就像是專案專屬的待辦事項清單。

### Your User Profile（個人檔案）
這是你的開發者名片！它會記錄你的貢獻點滴。你也可以做一個專屬的「個人 README」來介紹自己。

## ✍️ 在 GitHub 上使用 Markdown

在寫 Issue 或 PR 時，你可以用 Markdown 語法來排版。這是一種簡單的寫法，可以讓你輕鬆做出標題、粗體、清單，甚至插入 GIF 動圖和圖片，讓內容更易讀、更有趣！

## 🌍 加入 GitHub 社群

GitHub 不只是工具，它更是一個巨大的社群：

- **Star（給顆星）：** 看到喜歡的專案就點星星收藏，這也代表你對作者的鼓勵。
- **Follow（關注）：** 追蹤厲害的大神，你就能在首頁看到他們最近在忙什麼酷東西。
- **Explore（探索）：** 去 [github.com/explore](https://github.com/explore) 逛逛，發現全球最熱門的技術趨勢。

## ❓ 學完之後的疑問：Fork 跟 Clone 有什麼不同？

這兩個詞聽起來都是「複製」，但目的和存放位置完全不同：

| 特性 | Fork（雲端複製） | Clone（下載到電腦） |
|------|----------------|------------------|
| 存放位置 | 仍在 GitHub 雲端 | 在你的個人電腦（本地） |
| 主要目的 | 想修改別人的專案並貢獻（PR） | 準備在電腦上寫程式、改檔案 |
| 關聯性 | 與原專案保持連結，可接收更新 | 只是單純的檔案下載與同步 |
| 操作權限 | 不需要原作者權限即可執行 | 通常需要權限才能同步回雲端 |

## 💬 給老師的補充

透過這個 GitHub 工作流程的學習，也看了老師分享的相關連結，從裡面發現一個可以很快上手的圖形化介面網站：[Learn Git Branching](https://learngitbranching.js.org/?locale=zh_TW)。

對於比較善於圖像思考的設計師而言，在接觸工程師的世界時，這個網站相對友善很多。網站內會用引導式的方式，讓你直接操作打出程式碼，並在右側以圖像方式解釋運作原理，以便理解各式術語（如 Git Commit、Git Branch 等等）。

同時對於我不理解的地方，我也問了 Gemini，請他用白話的方式解釋給我聽。像是一開始不理解「Git 是在處理改動紀錄，而不是單純刪除檔案」以及「邏輯連動」的概念。

**Commit 雖是存檔，但跟設計師思維的存檔是不同概念。** 我們在資料夾裡看到的永遠是「同一個檔案」，但 Git 在背後幫你做了「時光堆疊」。

Gemini 用蓋房子的比喻解釋：

- Commit 1：地基（已存檔）
- Commit 2：牆壁（已存檔）
- Commit 3：屋頂（已存檔）

如果你突然覺得牆壁（Commit 2）畫錯了，你不是去資料夾刪掉一個叫做「牆壁」的檔案（因為它已經跟地基、屋頂連在一起了），而是跟 Git 說：「請幫我 Revert（撤銷）Commit 2。」Git 就會很聰明地把「蓋了牆壁」這個動作抽掉，但地基跟屋頂還會留著。

這也引發了我的另一個疑問：當牆壁被抽掉，地基跟屋頂是連動的，不會導致程式碼有問題嗎？Gemini 的解答讓我完全理解了 Commit、Revert 的底層邏輯——

**重點筆記：Git 的時間線是「往前跑」的。** 你並不是真的「回到過去」，而是：

- Commit 4（Revert 2）：撤銷牆壁（新存檔，此時屋頂會懸空報錯）
- Commit 5：修正邏輯（新存檔，補上柱子，或把屋頂接回地基）

這些我發現透過線上資源＋實作練習就可以很快理解。讀很多文字乍看感覺好像懂了，但實作才會發現真正不懂的地方在哪裡！
